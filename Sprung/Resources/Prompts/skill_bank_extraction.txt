## Task

Extract a COMPREHENSIVE skill inventory from this document. This is a BANK of
all possible skills to draw from at resume time — completeness matters more
than selectivity. When in doubt, INCLUDE the skill.

### What Counts as a Skill

**Explicitly demonstrated:**
- Programming languages (Python, Swift, C++)
- Frameworks and libraries (React, FastAPI, PyTorch)
- Tools and platforms (Git, Docker, AWS, SolidWorks)
- Hardware (Arduino, Raspberry Pi, PLCs, oscilloscopes)
- Fabrication (CNC machining, 3D printing, soldering, welding)
- Scientific methods (spectroscopy, microscopy, statistical analysis)
- Domain expertise (liquid crystals, embedded systems, machine learning)
- Methodologies and processes (Agile, experimental design, peer review, design thinking)
- Writing and communication (technical writing, grant writing, public speaking, peer-reviewed publication)
- Research methods (literature review, experimental design, data collection, statistical analysis)
- Regulatory/compliance (ITAR, ISO, FDA, OSHA, IRB)
- Soft skills only if DEMONSTRATED (led team of 5, mentored interns)

**Implied by context — extract these too:**
- If someone writes a SolidWorks design doc, they also have: 3D CAD, parametric modeling,
  engineering drawings, BOM management
- If someone publishes a paper, they also have: academic writing, peer review process,
  literature review, citation management
- If someone manages a lab, they also have: safety protocols, equipment maintenance,
  inventory management, student supervision
- If someone uses Arduino, they also have: embedded C/C++, serial communication,
  digital/analog I/O, circuit prototyping

Extract both the explicit skill AND its reasonable implied skills — with signal
discipline:
- Set `implied` to true on implied skills (and false on explicitly demonstrated
  ones); the evidence `context` should explain what the inference rests on
  (e.g., "SolidWorks design doc presumes engineering drawing practice")
- Cap implied-skill evidence strength at "supporting" or "mention" — NEVER "primary"
- An implied skill the applicant couldn't speak to in an interview is a "mention"
  at most; if even that feels like a stretch, omit it

### Evidence Requirements

Every skill should reference this document, but keep evidence LIGHTWEIGHT to
maximize the number of skills extracted:
- **location**: {LOCATION_GUIDANCE}
- **context**: Brief note on how skill appears (5-20 words is fine)
- **strength**: primary (deep use), supporting (significant), mention (referenced or implied)

DO NOT skip a skill because you can't write a detailed evidence entry.
A brief mention is valid evidence. The downstream system uses evidence strength
to rank skills — light evidence at "mention" strength is far more useful than
omitting the skill entirely.

### ATS Variants

Include common variations employers might search for:
- "Python" → ["python", "Python 3", "python3", "Python programming"]
- "Machine Learning" → ["ML", "machine learning", "deep learning", "neural networks"]
- "SolidWorks" → ["solidworks", "SolidWorks CAD", "3D CAD"]

ATS variants are VARIANTS of one skill, NEVER separate skill entries. Merge
trivially-identical variants at extraction time: if you find yourself emitting
"Python" and "Python 3" as two skills, collapse them into one canonical entry with
the variants listed in `ats_variants`. The same applies to abbreviation pairs
("ML" / "Machine Learning") and capitalization or spelling variants — one entry,
never near-duplicates.

### Categories

Assign each skill to ONE category. Start with these universal anchors (available for any profession):
- Tools & Software
- Leadership & Management
- Communication & Writing
- Methodologies & Processes

Then propose 2-5 additional domain-appropriate categories that fit the user's profile.
For example, a physicist might use "Hardware & Electronics" and "Scientific & Analysis";
a nurse might use "Clinical Skills" and "Patient Care"; a software developer might use
"Programming Languages" and "Frameworks & Libraries".

Keep total categories between 6-10 to avoid over-fragmentation.

### Quality Checklist

- Exhaustiveness: Could someone reading the source document identify skills you missed?
- Implied skills: Did you extract skills implied by the tools, roles, and activities described?
- Implied skills have `implied: true` and are capped at "supporting"/"mention" evidence strength
- ATS variants cover common job listing phrasings — and stay inside `ats_variants`, never as separate entries
- No near-duplicate entries: trivially-identical variants merged into one skill at extraction time
- Categories are appropriate
